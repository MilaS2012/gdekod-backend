# Тесты (план)

Сейчас тестов нет — папка-заглушка. Договорённости по будущему тестированию.

## Что и как тестируем

| Слой | Тип теста | Инструмент | Где запускается |
|---|---|---|---|
| `src/utils/` (cors, response) | Юнит, чистые функции, без I/O | Vitest | Локально + GitHub Actions |
| `src/db/queries/` | Интеграционный, с реальной PG | Vitest + Testcontainers (PG в Docker) | Локально + GitHub Actions |
| `src/handlers/public/` | E2E, вызываем handler как функцию с моком event/context | Vitest | Локально + GitHub Actions |
| Деплой на YC | Smoke: `curl` к URL функции после деплоя | GitHub Actions step | GitHub Actions |

## Почему Vitest, а не Jest

- ESM из коробки (мы на `"type": "module"`); у Jest до сих пор ESM в beta.
- Быстрее cold-start.
- Совместимый Jest-стайл API (`describe`, `it`, `expect`).
- Пишут активно, экосистема жива.

## Что не покрываем тестами

- **Сам PostgreSQL.** Это чужой код, доверяем.
- **`pg` (node-postgres).** Тоже чужой. Мокаем pool в integration-тестах, если нужно — но лучше Testcontainers, см. выше.
- **CORS-поведение браузера.** Тестим только что заголовки правильные; что браузер их корректно интерпретирует — гарантия браузера.

## Когда заводить

После первого успешного деплоя `merchants` и `coupons` на YC и проверки, что эндпоинты отдают данные. Раньше — преждевременно: API ещё может поменяться, тесты придётся переписывать.

Ориентир: **до первого подключения публичного фронта `gde-code.ru` к Yandex-эндпоинтам.** Без тестов туда не выкатываемся.
